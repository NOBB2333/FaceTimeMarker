# 实施路线图

最后更新：2026-07-07

## 路线图原则

1. 先把当前闭环做稳，再扩新模型。
2. 先评测识别质量，再凭感觉调参。
3. 先补数据模型，再堆 UI。
4. 本地单机体验优先，云端化不是当前目标。

## P0：稳定当前体验

目标：减少“看不到、点不到、找不到、生成到哪了”的体验问题。

任务：

- [x] 页面改成真实路由：`/workspace`、`/profiles`、`/search`。
- [x] 人物档案页拆分组件，降低页面文件体积。
- [x] 右侧人物片段预览固定在顶部可见区域。
- [x] 四视图资产支持删除。
- [x] 人物档案支持重命名、合并、回收站。
- [x] 工作台任务列表显示耗时、状态和取消。
- [x] 时间轴支持缩放和平移。
- [ ] 工作台继续拆分：优先拆 timeline/selection/drag 逻辑。
- [ ] 搜索页上传图预览和空状态继续打磨。

## P1：识别质量评测

目标：把“准不准”从主观感受变成可比较数据。

任务：

1. 建立样例集：
   - 真人清晰。
   - 真人低清。
   - 动漫清晰。
   - 动漫低清。
   - 多人同屏。
2. 每个样例记录：
   - 视频时长。
   - 分辨率。
   - 采样帧率。
   - raw detections。
   - kept faces。
   - tracks。
   - people。
   - 误检数。
   - 漏掉主角数。
   - 总耗时。
3. 输出 benchmark 报告。
4. 固化推荐配置：
   - 默认。
   - 动漫高召回。
   - 动漫低清严格。

## P2：人物“形象”模型

目标：解决一个人物有多套服装/造型、多张四视图资产的问题。

建议模型：

```text
global_person
  -> appearance
       -> reference_candidates
       -> global_person_four_view_assets
       -> notes
```

任务：

- [ ] 新增 appearance 表。
- [ ] 四视图资产绑定 appearance，而不只是 global_person。
- [ ] 按作品模式下可把某个人的参考帧新建为某个形象。
- [ ] 支持把错误形象合并到另一个形象。
- [ ] 人物档案按人物查看时，先看形象，再看作品观测。

## P3：源媒体重链接

目标：默认不复制源文件，但源文件移动后能恢复。

任务：

- [ ] 数据库记录 `source_path`、`current_path`、`missing` 状态。
- [ ] 播放/截帧失败时标记源媒体丢失。
- [ ] 前端显示“源媒体丢失，请重新链接”。
- [ ] 用户选择新路径后更新视频路径。
- [ ] 保留原始文件名和来源目录。

## P4：搜索质量

目标：让以图搜图结果可解释、可调、可用。

任务：

- [ ] 上传图明确预览。
- [ ] 阈值滑杆和数值输入。
- [ ] 结果显示相似度、质量标记、观测数量。
- [ ] 空结果解释“可能原因”和下一步。
- [ ] 点击结果跳转人物档案或工作台。
- [ ] 对明显低质量上传图给提示。

## P5：数据迁移和 ORM 清理

目标：保持数据库访问统一，未来切数据库不重写业务。

任务：

- [ ] 清理业务层残留手写 SQL。
- [ ] 所有读写走 ResultStore/SQLAlchemy。
- [ ] Alembic migration 覆盖新增表字段。
- [ ] 增加旧库升级测试。
- [ ] 明确哪些功能依赖 SQLite 特性，避免无意绑定。

## P6：更高级识别能力

只有 P1 评测证明当前模型无法满足时再做。

候选方向：

- 动漫专用人脸检测模型。
- 动漫人脸 embedding 模型。
- 手动画脸并补录 detection/track。
- 参考素材人工强制指定人物。

暂不优先：

- 人体框模型。
- 姿态估计。
- 全身分割。

## 验收标准

每个阶段完成时至少满足：

| 项目 | 标准 |
| --- | --- |
| 构建 | `pnpm --dir web build` 通过 |
| 后端 | 涉及 API/存储时 pytest 通过 |
| 文档 | README 或 docs 已同步 |
| 体验 | 新增危险操作有确认，长列表可滚动 |
| 数据 | 新字段有迁移/兼容策略 |
