# 需求与技术选型

最后更新：2026-07-07

## 需求定义

FaceTimeMarker 是以“人物脸部身份”为核心的视频分析工具，不是通用目标检测项目。

必须解决：

| 需求 | 说明 |
| --- | --- |
| 单视频人物识别 | 输出视频内人物、代表脸、出现片段 |
| 同人身份保持 | 同一个人物多次出现时尽量合并 |
| 人物时间轴 | 按人物轨道展示片段并可跳转播放 |
| 人工修正 | 命名、删除误检、合并、拖拽归类 |
| 跨视频人物档案 | 多部作品里把同一人物归到一个档案 |
| 参考素材 | 导出完整画面、脸部标记、脸部截图 |
| 四视图资产 | 用参考帧生成或上传角色设定图 |
| 本地配置 | 识别、输出、AI 参数统一写 TOML |

非目标：

- 不做完整剪辑软件。
- 不做云端协作。
- 不做训练集标注平台。
- 暂不做人形框模型。

## 推荐主线

```text
InsightFace + BoxMOT BoT-SORT + HDBSCAN + SQLAlchemy + FastAPI + Vue
```

| 环节 | 方案 | 原因 |
| --- | --- | --- |
| 视频读取 | OpenCV | 稳定、易截帧、易导出 |
| 人脸检测 | InsightFace SCRFD | 成熟，能给 bbox 和关键点 |
| 人脸特征 | InsightFace ArcFace | 身份相似度判断稳定 |
| 短期跟踪 | BoxMOT BoT-SORT | 连续帧 ID 稳定性较好 |
| 长期聚类 | HDBSCAN | 不需要预先知道人数 |
| 固定人数 | KMeans | 用户明确填人数时使用 |
| 数据库 | SQLAlchemy + SQLite 默认 | 本地零部署，同时保留切库能力 |
| API | FastAPI | 本地前后端分离和脚本调用 |
| 前端 | Vue 3 + TypeScript + Tailwind | 工具型工作台可控 |

## 为什么不加人体模型

当前多模态生成四视图的需求是“让 AI 知道目标人物是谁”。用脸部红框标记完整画面已经能满足：

```text
完整画面 + 脸部红框 -> AI 能知道目标角色和上下文
```

暂不加人体模型的原因：

1. 多一个检测模型会增加安装、算力和配置成本。
2. 动漫人体框也会受画风影响，不一定比脸框稳定。
3. 当前重点是人物身份和参考素材，不是精确人体分割。
4. 如果后续确实需要整身裁剪，可以作为新的 `ReferenceExporter` 模式，而不是改主线识别。

## 为什么需要脸部标记

只给完整画面时，画面里可能有多个人，AI 不一定知道目标是谁。

只给脸部截图时，又缺少服装、身体和场景。

所以参考素材保留三种视图：

| 视图 | 解决的问题 |
| --- | --- |
| 完整画面 | 身体、服装、环境和多人关系 |
| 脸部标记 | 明确目标人物 |
| 脸部截图 | 身份快速比对 |

## 聚类方案

| 方案 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| KMeans | 简单、快 | 需要预设人数 | 只在用户填人数时用 |
| DBSCAN | 不用预设人数 | `eps` 敏感 | 保留备选 |
| HDBSCAN | 适合密度不均，不用预设人数 | 参数更多，会有噪声 | 默认 |

动漫素材常见误差：

- 同一个角色不同表情/侧脸被分裂。
- 背景纹理被识别人脸。
- 短片段人物被当噪声。

因此需要通过配置预设和人工整理共同解决，而不是只靠一个聚类算法。

## 跟踪方案

| 跟踪器 | 适配度 | 说明 |
| --- | --- | --- |
| BoT-SORT | 高 | 当前默认，适合用检测框和外观特征保持短期 ID |
| ByteTrack | 中 | 快，但默认更偏检测框关联，保留参考 |
| DeepSORT | 中低 | 老方案，概念清楚但不是首选 |
| DeepOCSORT/BoostTrack | 待评估 | 后续可作为新 adapter |

跟踪器只解决连续片段，不解决十分钟后再次出现是不是同一个人。长期身份仍依赖人脸 embedding 和聚类。

## 数据库选择

默认 SQLite，因为：

- 本地零部署。
- 单人使用足够。
- 方便打包、迁移和备份。

但代码不应写死 SQLite。当前约定：

```text
业务逻辑 -> ResultStore -> SQLAlchemy ORM -> 数据库
```

未来切 PostgreSQL/MySQL 时，优先完善：

1. SQLAlchemy 模型。
2. Alembic migration。
3. 测试用例。
4. 配置中的数据库 URL。

## 配置选择

用户明确要求不依赖环境变量，因此当前设计允许把 API Key 和模型参数直接放在项目配置里。

优点：

- 所有配置集中可读。
- 本地使用直接。
- 不需要额外部署环境。

风险：

- 如果项目公开，可能泄露 Key。

处理：

- 文档中提醒私有使用。
- 不在文档里抄写具体 Key。
- 后续可以提供 `.example.toml` 或“本地覆盖配置”方案，但不强制。

## 动漫低清调参策略

| 目标 | 配置倾向 |
| --- | --- |
| 尽量找全主角 | `anime-high-recall.toml` |
| 减少背景误检 | `anime-lowres-strict.toml` |

关键参数：

| 参数 | 调高后 | 调低后 |
| --- | --- | --- |
| 采样帧率 | 更准、更慢、红框更新更密 | 更快、可能漏短片段 |
| 最小人脸尺寸 | 减少小纹理误检 | 更容易找远景小脸 |
| 检测置信度阈值 | 更严格、误检少 | 召回高、误检多 |
| 最小连续帧数 | 过滤闪现误检 | 保留短暂人物 |
| 丢失轨迹缓冲帧数 | 遮挡后更连续 | 减少串人 |
| 最小匹配阈值 | 减少串人 | 减少断轨 |

## 四视图生成选择

使用 LiteLLM 兼容图像接口，而不是直接绑定某一家平台：

- 兼容中转站。
- 可替换模型名。
- 请求参数集中在 `configs/ai.toml`。

资产单位是一张未切分原图，而不是四张独立图片。当前推荐横向角色设定表：左侧放正面全身、侧面全身、背面全身、45 度全身或半身，右侧做两列细节区，分别补 45 度脸部/轻微微表情和服装/发型/配饰细节。前端预览时不强行切图。
