# 项目架构

最后更新：2026-07-07

## 项目目标

FaceTimeMarker 是一个本地视频人物分析和人物档案系统：

```text
输入视频
  -> 人脸检测
  -> 连续帧跟踪
  -> 人脸特征聚类
  -> 人物出现时间轴
  -> 人物档案和参考素材
```

核心目标不是“检测到所有脸”本身，而是身份保持：同一个人在不同时间段、不同视频里出现时，最终能归到可整理的人物档案。

## 为什么不直接改上游项目

| 上游 | 角色 | 本项目用法 |
| --- | --- | --- |
| InsightFace | 人脸检测和 embedding | 通过 PyPI 包调用 |
| BoxMOT | 多目标跟踪算法集合 | 默认使用 BoT-SORT |
| ByteTrack | 跟踪算法参考 | 作为 Git 子模块参考，不参与默认运行 |

业务逻辑放在 `src/` 和 `web/`，不要改上游源码。这样升级、替换和调试都更可控。

## 根目录结构

```text
configs/                  TOML 配置，包含识别、输出、参考图、AI
data/                     上传暂存和本地输入
docs/                     文档和截图
_outputs/                 默认输出、数据库、人物库
scripts/                  启动、构建、检查、benchmark 脚本
src/                      Python 后端、CLI、分析管线
web/                      Vue 前端工作台
third_party/              上游源码参考子模块
tests/                    后端测试
migrations/               Alembic 迁移脚本
```

## Python 分层

```text
src/apps/
  cli.py                  CLI: inspect/analyze/benchmark/import/serve/hardware/people
  api/main.py             FastAPI: 视频、任务、人物、搜索、四视图

src/core/
  config.py               配置模型和中文配置键
  hardware.py             ONNX Runtime provider 探测
  pipeline.py             分析主流程
  cache.py                检测/跟踪缓存
  models.py               核心数据模型
  storage/
    orm_models.py         SQLAlchemy ORM 模型
    database.py           会话和数据库连接
    sqlite_store.py       ResultStore 实现和业务读写
  timeline/               轨迹到时间轴、上下文片段
  clustering/             HDBSCAN/DBSCAN/KMeans 相关封装

src/adapters/
  face/                   InsightFace 适配器
  tracking/               BoT-SORT / ByteTrack 适配器
  video/                  OpenCV 视频读取
  exporters/              JSON、CSV、截图、片段、参考图导出
  identity/               跨视频人物库
```

分层原则：

- `core` 不直接绑定具体 UI。
- `adapters` 包装外部库。
- `apps` 负责 CLI/API 协议，不承载复杂算法。
- 数据读写集中到存储层，不在路由和前端拼 SQL。

```text
web/src/app/pages.ts             页面配置

  WorkspacePage.vue              工作台页面容器

```

当前路由：

```text
/workspace
/profiles
/search
```

这样刷新页面不会回到默认页，也方便以后给搜索结果、人物档案详情做深链接。

## 配置结构

```text
configs/default.toml
  -> recognition.toml
  -> outputs.toml
  -> reference_export.toml
  -> ai.toml

configs/profiles/
  anime-high-recall.toml
  anime-lowres-strict.toml
```

配置原则：

1. 常用配置必须有中文注释。
2. 单视频特殊参数用 profile 配置覆盖。
3. 识别参数变化通常需要重新识别。
4. AI 配置变化不应触发视频重新分析。

## 数据和输出

默认输出：

```text
_outputs/
  facetimemarker.db
  people_index.json
  <video>/
    timeline.json
    timeline.csv
    faces/
    references/
    clips/
```

源媒体策略：

- 默认不复制源文件，数据库记录源路径。
- 浏览器上传无法保留本机真实路径，只能保存到 `data/uploads/`。
- 如果源媒体丢失，后续应做“重新链接源媒体”能力。

## 数据库架构

当前默认 SQLite，但架构目标不是被 SQLite 绑定死。

```text
业务代码 -> ResultStore -> SQLAlchemy ORM -> SQLite / 其它数据库 URL
```

主要领域：

| 领域 | 表 |
| --- | --- |
| 视频 | `videos` |
| 本地人物 | `people`, `segments`, `tracks`, `track_detections`, `face_crops` |
| 跨视频人物 | `global_people`, `global_observations`, `global_person_actions` |
| 人工整理 | `review_actions` |
| 四视图 | `global_person_four_view_assets` |
| 任务 | `analyze_jobs` |

后续切数据库时，应优先完善 ORM 和迁移，而不是在业务层写数据库专用 SQL。

## 核心流程

```text
video path
  -> VideoReader
  -> FaceAnalyzer
  -> Tracker
  -> Clusterer
  -> TimelineBuilder
  -> Exporters
  -> ResultStore
```

可替换点：

| 接口 | 当前默认 | 未来可替换 |
| --- | --- | --- |
| FaceAnalyzer | InsightFace | 动漫人脸检测/特征模型 |
| Tracker | BoxMOT BoT-SORT | DeepOCSORT、BoostTrack、ByteTrack |
| Clusterer | HDBSCAN | DBSCAN、Agglomerative、人工固定分组 |
| Storage | SQLite via SQLAlchemy | PostgreSQL/MySQL |
| Image generation | LiteLLM compatible | OpenAI 官方或其它兼容接口 |

## 当前实现状态

1. CLI、FastAPI、Vue 工作台已形成闭环。
2. 分析任务支持单视频、批量队列、状态列表和取消。
3. 数据库保存视频、人物、片段、轨迹、逐帧框、截图、人物档案、四视图资产。
4. 前端有工作台、人物档案、搜索三个页面。
5. 人物档案支持新建、重命名、合并、回收站、四视图上传/生成/删除。
6. 搜索支持文本和以图搜图。
7. 配置已经拆分，支持单视频配置覆盖。

## 架构边界

当前不做：

- 云端多用户权限。
- 完整训练集标注平台。
- 自动人体框模型。
- 在线协作。
- 大规模向量数据库。

这些以后可以做，但不应影响当前本地离线闭环的稳定性。
